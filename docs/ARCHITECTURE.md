# Архитектура панели управления игровыми серверами

## 1. Общая схема

```
                      ┌───────────────────────────┐
                      │        Browser (SPA)      │
                      │  React + Vite + TS + TW   │
                      └─────────────┬─────────────┘
                                    │ HTTPS REST + WSS
                                    ▼
                      ┌───────────────────────────┐
                      │        PANEL (master)      │
                      │  Go · gin/chi · JWT · RBAC │
                      │  PostgreSQL   Redis        │
                      └─────────────┬─────────────┘
                                    │ gRPC (mTLS) + WSS (реалтайм)
                     ┌──────────────┼──────────────┐
                     ▼              ▼              ▼
              ┌────────────┐ ┌────────────┐ ┌────────────┐
              │  Node #1   │ │  Node #2   │ │  Node #N   │
              │  wingsd    │ │  wingsd    │ │  wingsd    │
              │  (daemon)  │ │  (daemon)  │ │  (daemon)  │
              │  Docker API│ │  Docker API│ │  Docker API│
              └────────────┘ └────────────┘ └────────────┘
```

Панель (`panel`) — единая точка входа для пользователей и API. Она **никогда**
не трогает Docker напрямую — все операции с контейнерами делегируются демону
(`wingsd`), который развёрнут на каждой ноде. Панель и демон общаются по
защищённому каналу (gRPC + токен ноды, поверх TLS); браузер никогда не
обращается к демону напрямую — панель проксирует консоль/статистику через
свой собственный WebSocket-шлюз. Это отличает решение от Pterodactyl (там
Wings достижим напрямую) и упрощает модель безопасности: наружу торчит
только панель.

## 2. Структура репозитория

```
/
├── backend/                 # Панель-мастер (Go)
│   ├── cmd/panel/main.go     # точка входа
│   ├── internal/
│   │   ├── config/           # чтение ENV/конфига
│   │   ├── db/                # пул pgx, миграции
│   │   ├── models/            # доменные структуры (User, Node, Server, Egg...)
│   │   ├── auth/              # JWT, хэши паролей, RBAC middleware
│   │   ├── api/                # HTTP-роутер + handlers (REST v1)
│   │   ├── daemonclient/      # клиент для вызова wingsd на нодах
│   │   └── ws/                 # WS-шлюз: проксирует консоль/метрики в браузер
│   └── migrations/            # SQL миграции (golang-migrate формат)
│
├── daemon/                   # Wings-like демон, разворачивается на каждой ноде
│   ├── cmd/wingsd/main.go
│   ├── internal/
│   │   ├── config/            # токен ноды, TLS, docker socket
│   │   ├── docker/            # создание/старт/стоп контейнеров, cgroups-лимиты
│   │   ├── api/                 # HTTP+WS API, которым управляет панель
│   │   └── console/            # хаб стриминга stdout/stdin контейнера
│   └── proto/daemon.proto     # контракт gRPC (см. docs/PROTOCOL.md)
│
├── frontend/                 # SPA (React + Vite + TS + Tailwind)
│   └── src/
│       ├── styles/panel.css   # предоставленный дизайн, подключается как есть
│       ├── api/                 # fetch-клиент к backend REST/WS
│       ├── types/               # общие TS-типы (зеркало Go моделей)
│       ├── components/          # ServerCard, StatusBadge, ResourceBar...
│       └── pages/                # Dashboard, ServerView, Settings...
│
└── docs/                      # архитектурная документация (этот каталог)
    ├── ARCHITECTURE.md
    ├── DATABASE.md
    └── PROTOCOL.md
```

**Почему так:** `backend` и `daemon` — это два независимых Go-модуля
(отдельные `go.mod`), потому что деплоятся они на разные машины и не должны
тянуть общие зависимости (у демона нет и не должно быть доступа к
PostgreSQL). Общий контракт между ними — только `daemon/proto/daemon.proto`
(плюс сгенерированные из него типы, которые импортируют оба модуля через
общий `proto`-пакет либо публикуемый как отдельный Go-модуль
`github.com/yourorg/panel-proto`).

## 3. Роли компонентов

| Компонент | Отвечает за | Не отвечает за |
|---|---|---|
| **panel** (backend) | Auth/RBAC, CRUD нод/серверов/пользователей, оркестрация (кому какую ноду выбрать), хранение состояния, WS-прокси для браузера | Прямую работу с Docker |
| **wingsd** (daemon) | Docker API, cgroups-лимиты, файловый менеджер на диске сервера, консоль контейнера, установка (install script) | Аутентификацию пользователей, хранение "правды" о сервере (он — исполнитель команд панели, а не источник истины) |
| **frontend** | SPA, дашборды, файловый редактор в браузере, консоль (xterm.js) | Бизнес-логику — только вызывает REST/WS панели |

## 4. Масштабируемость

- Панель — стейтлесс (сессии в Redis, а не в памяти процесса) → можно
  поднимать несколько реплик за балансировщиком.
- Каждая нода регистрируется в панели (см. `POST /api/v1/nodes`), получает
  `daemon_token`; wingsd использует токен для аутентификации при обратном
  подключении к панели (long-lived gRPC stream) — это позволяет нодам стоять
  за NAT без входящих портов на панель.
- Redis также используется как pub/sub шина для рассылки live-метрик всем
  открытым WS-соединениям браузеров без похода в БД.

## 5. Безопасность

- JWT (access + refresh) для пользователей SPA; RBAC — роли и permissions
  хранятся в БД, проверяются middleware на каждый handler.
- Токен ноды (`daemon_token`) — генерируется один раз при добавлении ноды,
  хранится в БД только как bcrypt-хэш; демон предъявляет его как
  `Authorization: Bearer <token>` в gRPC metadata / HTTP header.
- SSH-ключи пользователей хранятся в таблице `ssh_keys` и используются для
  доступа к SFTP-серверу, встроенному в `wingsd` (см. `docs/PROTOCOL.md`).
  Логин — `<username>.<server-id>` на порту 2022; на каждую попытку входа
  `wingsd` дёргает `POST /api/v1/internal/sftp/authenticate` на панели,
  предъявляя свой `daemon_token`, и получает в ответ разрешение/запрет,
  `read_only` и полный UUID сервера — сам демон не хранит ни ключей, ни
  списка разрешённых пользователей, вся авторизация остаётся в БД панели.
- Все секреты (пароли БД сервера, токены) хранятся хэшированными либо
  шифруются симметрично (AES-GCM) с ключом из ENV панели — открытым текстом
  в PostgreSQL не хранится ничего чувствительного.
