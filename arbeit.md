# Arbeit — что ещё можно поправить и добавить

Живой список находок: реальные баги/недоделки уже в коде, плюс идеи для
новых фич. Составлено по результатам сквозного аудита схемы БД и кода —
искал места, где что-то читается/пишется наполовину, где в схеме есть
колонка/таблица без единого использования в Go-коде, и где документация
разошлась с тем, что реально запущено.

Не дублирую то, что уже пофикшено в этой сессии (SFTP, бэкапы, домены,
suspend, capacity ноды, egg-валидация, создание/сброс пароля, TOCTOU-гонки
и т.д. — всё это уже в `add.md` и запушено).

## 1. Реальные баги / недоделки (стоит проверить и поправить)

### 1.1. Activity log жёстко обрезан на 100 записей, без пагинации
`ActivityHandler.List` делает `... LIMIT 100` и всё — ни `OFFSET`, ни
фильтра по дате, ни поиска по пользователю/событию. На активной панели
через какое-то время старые события становятся физически недоступны
через UI (они в БД лежат, но их никогда не покажут). Стоит добавить
пагинацию (`?before=<id>` или `?page=`) и, желательно, фильтр по
серверу/пользователю/типу события — благо `activity_logs` уже
проиндексирован под это (`created_at DESC`, `(server_id, created_at DESC)`).

### 1.2. `servers.status` в БД не обновляется вообще после создания
Уже находили и чинили симптом (застревание online/offline после обрыва
WS), но корень шире: колонка `status` в БД пишется РОВНО один раз — при
создании сервера (`'installing'` → `'offline'`). Все дальнейшие
"running"/"starting"/"stopping", которые видно в интерфейсе — чисто
фронтовая иллюзия поверх WS-статистики, которая живёт только в памяти
браузера и обнуляется при перезагрузке страницы (до следующего сообщения
по WS). Если сервер выключится, пока никто не смотрит на страницу — в
БД он навсегда останется "offline" с последнего раза, даже если через
минуту сам поднимется. Список серверов (`ServerList`) должен уметь это
показать корректно даже без открытой WS-сессии. Разумный фикс: демон
периодически (или сразу при поллинге статы) шлёт панели актуальный
`state`, а панель пишет его в `servers.status` — то есть `Hub.pollStats`
должен не только транслировать в браузер, но и апдейтить БД.

### 1.3. `nodes.agent_version` — колонка есть, никто её не пишет
Ни демон при старте/пинге не сообщает свою версию, ни панель её нигде
не читает. Было бы полезно (и совсем недорого сделать, раз версия
демона теперь честный semver — `daemon/cmd/wingsd` можно вкомпилировать
version так же, как `backend/cmd/panel` уже делает) — тогда на странице
Nodes видно, какие ноды отстали и надо обновить, вместо того чтобы
руками помнить.

### 1.4. CORS `AllowedOrigins` захардкожен на `http://localhost:5173`
`backend/internal/api/router.go`, `cors.Handler`. В проде панель и API
живут на одном домене через nginx, так что для обычной работы браузера
это не мешает (same-origin, CORS вообще не участвует). Но если
когда-нибудь понадобится браузерный доступ к API с другого домена
(например, отдельный дашборд/интеграция стороннего сайта через API-key
прямо из JS в браузере пользователя) — он молча упрётся в CORS. Стоит
либо вынести allowed-origin в конфиг (`PANEL_ALLOWED_ORIGINS`), либо
явно задокументировать, что API рассчитан только на server-to-server
использование ключей, а не на browser-to-API с чужого домена.

### 1.5. Список серверов (`ServerHandler.List`) без пагинации
Не критично при десятках серверов, но при сотнях/тысячах — это один
`SELECT *` без `LIMIT`. Не горит, но если панель когда-то вырастет,
стоит добавить `?limit=&offset=` заранее, а не когда страница начнёт
тормозить.

### 1.6. `docs/PROTOCOL.md` описывает gRPC, которого нет и не было
Документ явно расписывает control-plane на gRPC + mTLS, ссылается на
`daemon/proto/daemon.proto` (файл реально существует, 173 строки) — но
`google.golang.org/grpc` не подключен НИ В ОДНОМ модуле, никакого
сгенерированного `.pb.go` нет, вся реальная связь panel↔wingsd — это
обычный HTTP/JSON + WS с Bearer-токеном (то, что построено и работает
всю эту сессию). Похоже, gRPC был исходным планом, от которого тихо
отказались в пользу более простого HTTP-подхода, а документацию не
поправили. Стоит либо переписать `PROTOCOL.md` под факт (HTTP/JSON + WS,
как оно есть — эндпоинты уже все перечислены в `router.go` с обеих
сторон), либо, если gRPC зачем-то всё ещё в планах — явно пометить
раздел как "не реализовано, roadmap".

### 1.7. Разрешено ли двум людям одновременно писать в один файл?
`FileHandler.Write` просто перезаписывает файл без какой-либо проверки
конфликтов (нет `If-Match`/mtime-check). Два открытых редактора одного
файла — тихо потеряют правки друг друга (last write wins). Не
блокирующий баг, но стоит хотя бя предупреждать в UI, если файл
изменился на диске с момента открытия.

## 2. Что стоит добавить (новые возможности)

### 2.1. Экспорт/скачивание бэкапа панели целиком (не серверов, а самой БД)
Есть бэкапы СЕРВЕРОВ (файлы игровых серверов), но нет ничего для бэкапа
самой панели — `pg_dump` базы + `panel.env`/`wingsd.env` секретов. Если
диск с панелью умрёт, всё (пользователи, ноды, права, история) исчезает
без единого штатного способа восстановления. Даже простой
`scripts/panel-backup.sh` (cron + `pg_dump` в `/var/backups/panel`) закрыл
бы огромный риск для self-hosted инструмента, где нет какого-то
managed-провайдера, который сам всё бэкапит.

### 2.2. Email-уведомления (опционально, если будет SMTP)
Сейчас панель ничего никому не пишет: ни "твой сервер упал", ни "бэкап
не удался", ни "кто-то вошёл в твой аккаунт с нового IP". Для
self-hosted тула это осознанный трейд-офф (не поднимать почтовую
инфраструктуру), но если когда-то захочется — уже есть вся необходимая
активность в `activity_logs`, можно подключить простой SMTP-воркер,
который шлёт дайджест или алерты по важным событиям
(`server.power.kill` из-за OOM, `backup.*` неудача, `node.*` недоступна).

### 2.3. Массовые операции над серверами
Сейчас всё — по одному серверу за раз (start/stop/restart/backup).
Для админа с сотней серверов было бы полезно: выбрать несколько на
Dashboard и разом рестартовать/забекапить/приостановить. Не критично,
но ощутимо ускоряет жизнь на масштабе.

### 2.4. Собственный сертификат для конкретного узла заранее (staging)
Сейчас домен для сервера настраивается ПОСЛЕ создания сервера
(вкладка Domains), и certbot идёт синхронно в момент добавления —
если DNS ещё не разрулился, попытка проваливается и надо пересоздавать
домен заново. Можно сделать "отложенный" статус (`tls_status: pending`)
и фоновый ретрай certbot каждые N минут в течение часа-двух, вместо
одной попытки и всё.

### 2.5. Двухфакторка не обязательна даже для админов
Ни на уровне панели, ни на уровне отдельного пользователя нет опции
"обязательная 2FA для админов". Для панели, которая теперь управляет
SSH-ключами, доменами и бэкапами — это не помешало бы как минимум
опциональной политикой ("is_admin ⇒ обязать включить 2FA при следующем
входе").

### 2.6. Поиск и фильтры на страницах Users/Activity/Nodes
Простой `<input>` фильтр по имени/email/событию — сейчас на этих
страницах ничего, кроме сырого списка (в отличие от Dashboard, где
поиск серверов уже есть).
